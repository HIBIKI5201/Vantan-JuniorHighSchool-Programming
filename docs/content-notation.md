# 授業資料の記法ルール

このサイトが読み込む `src/content/courses/` と `src/content/lessons/` の中身をどう書くか、
というルールをまとめたもの。Notionから移行した過去の資料も、これから新しく書く資料も、
このルールに合わせる。

実際に移行済みの9コース分のファイルを見て書いているので、ここに書いてあるパターンは
全部 `src/content/lessons/` の中に実例がある。迷ったら実例を探すのが一番早い。

## ディレクトリ構成

```text
src/content/
├── courses/
│   └── <courseSlug>.md       # コース1つにつき1ファイル(例: kinyo-2026-1-3.md)
└── lessons/
    └── <courseSlug>/
        └── <NN>.md           # 授業回1つにつき1ファイル。NNは "00", "01"... の2桁ゼロ埋め
public/
└── lessons/
    └── <courseSlug>/
        └── <NN>/
            └── imageX.png     # そのレッスンで使う画像
_notion-source/                # Notionエクスポートの生データ置き場(.gitignore済み、コミットしない)
scripts/
└── migrate-from-notion.mjs    # _notion-source/ → src/content/ + public/lessons/ への変換スクリプト
```

`courseSlug` は既存のものは `<曜日ローマ字>-<年>-<開始月>-<終了月>` の形式に揃えている
(例: `kinyo-2026-1-3` = 金曜日, 2026年1〜3月期)。曜日を含まないコース(体験授業・Scratch wiki)は
`taiken-jugyo` / `scratch-wiki` のように内容が分かる短い英字にする。

## コースのfrontmatter (`src/content/courses/<slug>.md`)

```yaml
---
title: "金曜ゲーム"        # コース名(曜日・期間を含まない)
period: "2026 1~3月"       # 表示用の期間
dayOfWeek: "金曜日"        # 該当しない場合は "-" や "単発" など
order: 20                  # トップページでの並び順(大きいほど上に表示される)
status: complete           # "complete" | "partial"
note: "..."                # 任意。トップ/コース一覧ページの💡欄に表示される
---
```

本文は基本的に空でよい(トップページ・コース一覧ページはfrontmatterしか見ていない)。

### `order` の付け方(実際に使っている値)

`order` は日付順を厳密に守る必要はなく、「大きい数字ほど上に表示したい」という表示順の指定でしかない。
今使っている値:

| コース | order |
| :--- | :--- |
| Scratch wiki(リファレンス) | 1 |
| シューティングゲーム(2025 10~12月) | 5 |
| ブロック崩しゲーム(2025 10~12月) | 6 |
| 金曜ゲーム(2026 1~3月) | 20 |
| 火曜ゲーム(2026 1~3月) | 21 |
| 金曜ゲーム(2026 4~6月) | 22 |
| 金曜ゲーム(2026 7~9月) | 23 |
| 水曜日ゲーム(2026 7~9月) | 24 |
| 体験授業(単発・随時) | 30 |

新しいコースを追加する時は、時系列で一番新しいものが一番大きい数字になるように、
間を空けて(例: 25, 26...)採番する。体験授業のような単発コースは常に一番上(一番大きい数字)に
しておくと見つけやすい。

## レッスンのfrontmatter (`src/content/lessons/<courseSlug>/<NN>.md`)

```yaml
---
course: "kinyo-2026-1-3"   # 対応するコースのslug
order: 3                   # "#3" などの回数。ファイル名のNN(2桁ゼロ埋め)と対応させる
title: "#3 敵を作ろう"      # 一覧・ページタイトルに表示される
status: complete           # "complete" | "partial"
note: "..."                # 任意。status: partial の時にページ上部の💡欄に表示される
---
```

`sessionDate` のようなフィールドは今のところ使っていない。授業日の情報は本文中に
`授業日: 2026年2月6日` のようにテキストとしてそのまま書かれている(次の節を参照)。
将来的にfrontmatterに切り出したくなったら `src/content/config.ts` のschemaに追加すること。

### `status: partial` の使いどころ

コース・レッスンのどちらにも `status: partial` が使える。

- コースに付けると、トップページのカードに「一部準備中」バッジが付く。
- レッスンに付けると、コース一覧ページのそのレッスン名の横に「画像準備中」バッジが付き、
  レッスンページの本文の上に `note` の内容が💡ボックスで表示される。
- レッスンが1件もないコースは、コース一覧ページに
  「まだ授業回のページがありません。準備が整い次第、追加します。」と自動で表示される
  (`src/pages/courses/[course]/index.astro` の分岐処理)。

今のところ全9コース・全レッスンが `status: complete` になっている
(過去に画像なしの暫定版を`partial`で載せたことがあったが、後から画像付きの完全な
エクスポートが届いたので複製せず`complete`に置き換えた)。

## 本文の記法

本文はNotionのMarkdownエクスポートの書き方をほぼそのまま使う。

### 見出しレベル(2つの流儀が混在している)

コースが作られた時期によって、見出しの深さの使い方が2パターンある。**どちらも正しく表示されるので
無理に統一しない**。新しく書く時はどちらでもよいが、1つのレッスンの中では一貫させること。

- **新しい形式**(2026年7月〜の金曜/水曜ゲームなど): `#` はページタイトルのみ、
  `## 大セクション`、`### 手順の小見出し` という3段階。
- **古い形式**(2025〜2026年前半の金曜/火曜/シューティング/ブロック崩しなど): `#` を
  ページタイトルと大セクションの両方に使い(同じ階層)、`## 手順の小見出し` で1段だけ下げる、
  という2段階。

どちらの形式でも、`src/layouts/Layout.astro` のグローバルCSSが `h1`/`h2`/`h3` それぞれに
見た目を付けているので表示上の問題はない。

### よく出てくるセクション名(呼び方が時期によって揺れている)

新しく書く時は左の「推奨」を使うとよいが、右側の旧名称が出てきても、それは過去の資料の
書き方なので**リネームして回る必要はない**。

| 役割 | 推奨(新しいレッスンで使う) | 資料内で実際に使われている旧名称 |
| :--- | :--- | :--- |
| その回のゴール(箇条書き) | `目標` | (これは全時期共通) |
| 先生の完成品Scratchプロジェクトへのリンク | `見本` | `プロジェクト` |
| 事前に知っておくと良い知識・用語の解説(任意) | `覚えること` | `覚える事` |
| メインの制作パート | `やってみよう` | (古い形式では大セクション名がそのまま制作内容、例: `敵を作ろう`) |
| 締めの一言 | `終わり` | `終了`、`終了！！` |
| 追加でやってみる課題(任意) | `エクストラ課題` | `追加の実装` |
| その日の振り返りフォーム(任意、2026年7月〜の新しい回のみ) | `今日のひとことを書こう！`(Google Formの`forms.gle`リンク) | (旧形式にはなし) |

`見本`/`プロジェクト`セクションはURLをそのままの行に書くだけでよい(GitHub Flavored Markdownの
自動リンクが効くので `[リンク文字](url)` にする必要はない)。

`覚えること`セクションは任意で、算数・理科などプログラミング以外の前提知識を、図解や
参考動画・参考サイトへのリンクと一緒に説明する時に使う(例: `block-kuzushi-2025-10-12/01.md` の
「角度の考え方」)。

### 画像

```markdown
![image.png](/lessons/kinyo-2026-1-3/03/image.png)
```

- パスは必ず **`/lessons/<courseSlug>/<NN>/ファイル名`** というサイトルート相対パスにする。
  GitHub Pagesのbase path(`/Vantan-JuniorHighSchool-Programming`)は
  `astro.config.mjs` のremarkプラグインがビルド時に自動で付け足すので、**自分で書かない**。
- ファイル名の半角スペースはハイフンに置き換える(例: `image 1.png` → `image-1.png`)。
  これはURLにスペースが入る事故を防ぐため。
- 画像の実体は `public/lessons/<courseSlug>/<NN>/` に置く。
- **他のコース/レッスンの画像を再利用することもできる**(例: `scratch-wiki`の解説ページが
  `suiyo-2026-7-9`のレッスンのスクリーンショットを流用している)。その場合は
  `public/lessons/<借りる側のcourseSlug>/<NN>/` に画像をコピーしてから参照する
  (`migrate-from-notion.mjs`は元のNotionの相対パス `../他のフォルダ/画像.png` を自動で
  解決してコピーしてくれる。詳しくはスクリプト内の`relativeResolved`の処理を参照)。

### コールアウト(吹き出し)

```markdown
<aside>
💡

説明文1行目。

説明文2行目(段落を分けたい時は空行を入れる)。

</aside>
```

- 1行目は絵文字(基本は💡)。2行目以降が本文。
- `Layout.astro` のグローバルCSSが `aside` タグを自動でカード風にスタイリングするので、
  クラス名などは付けなくてよい。
- **書く内容のトーン**(中学生向けであることを踏まえて):
  - です・ます調で、生徒に語りかけるように書く(「〜しましょう」「〜ですね」)。
  - 新しく作る変数名・ブロック名は、英単語の意味を添えて説明する
    (例: 「名前は"MoveSpeed"にします。意味はそのまま「移動のスピード」ですね。」)。
  - 「なぜこの操作が必要か」を先に一言添えてから、具体的な手順の説明に入る。
  - 「こうすると〜になります」という因果関係の説明を使うと分かりやすい。
  - 詳しい手順は aside の外に番号付きリスト(`1.` `2.` ...)として書く。asideの中に手順を
    書かない(asideは「なぜ・何をするか」の説明、外の番号リストが「具体的な操作」という役割分担)。

### リストの使い分け

- `目標`セクションや、対頂角・同位角・錯角のような単なる列挙 → `- ` の箇条書き
- 実際にScratchで操作する手順 → `1.` `2.` の番号付きリスト(常にaside/画像の直後に置く)

### 内部リンクについて

Notionのエクスポートには、他ページへのリンク(`[変数とは：変数](../Scratch wiki/変数 xxxx.md)`)が
含まれることがある。**Scratch wikiの用語ページへのリンクだけは特別扱いされ、実際のサイト内リンク
(`/courses/scratch-wiki/<NN>/`)に変換される**(`scripts/migrate-from-notion.mjs` の
`SCRATCH_WIKI_LESSONS` マップを使ってファイル名から該当ページを特定している)。表示は
「📖 用語名」という丸いピル型のリンクになる(`Layout.astro` の
`article a[href*="/courses/scratch-wiki/"]` のスタイル)。

Scratch wiki以外の内部ページへのリンク(他コースの別レッスンへのリンクなど)は、このサイトでは
解決できないため、リンクを外してテキストだけ残すようになっている。手で新しく書く場合、
Scratch wikiの用語を参照したい時は `[用語](/courses/scratch-wiki/<NN>/)`
の形でリンクを書けばよい(NNは `scratch-wiki` コース内のそのレッスンの番号。
`src/content/lessons/scratch-wiki/` を見て確認する)。それ以外の内部リンクは書かず、
プレーンテキストにしておく。

(画像は前述の通り他コースのものも参照できる。)

## 画像が見つからない場合の表記

Notionのエクスポートに画像の実体が含まれていなかった場合は、壊れた画像リンクにする代わりに
次の形式のテキストを入れる(`scripts/migrate-from-notion.mjs` が自動でこの形にする):

```markdown
*[画像が見つかりません: image 2.png（元のNotionエクスポートに含まれていませんでした）]*
```

## Notionから新しいコース/レッスンを取り込む・更新する手順

1. Notionで対象のページを開き、「•••」→ エクスポート → Markdown & CSV
   → **「サブページを含む」に必ずチェックを入れて**書き出す
   (チェックを忘れると画像やレッスン本文が抜け落ちる)。
2. 展開したフォルダを `_notion-source/` 以下に置く。
   **フォルダの階層はNotionのエクスポートごとに微妙に変わることがある**
   (`プライベート、シェア/バンタン中等部/...` の場合と
   `プライベート、シェア/バンタン/バンタン中等部/...` の場合があった)。
   決め打ちで「中身がない」と判断せず、`_notion-source/` の下を実際にたどって
   本当のフォルダ名・階層を確認してから次に進むこと(過去に階層を1段見誤って、
   実在するコースを「未移行」と誤判定したことがある)。
3. `scripts/migrate-from-notion.mjs` の `COURSES` 配列に、対象コースの
   `srcFolder`(`_notion-source/`直下でのフォルダ名) ・ `slug` ・ `title` ・
   `dayOfWeek` ・ `period` ・ `order` を追加する。
4. `node scripts/migrate-from-notion.mjs` を実行する。
   コンソールに `画像が見つかりません` という警告が出たら、Notion側の元データに
   その画像が含まれていなかったということなので、無視してよい
   (本文には自動でプレースホルダーが入る)。
5. `npm run dev` でプレビューして、レッスン一覧・画像・コールアウトの表示を確認する。

このスクリプトは `#N タイトル.md` という命名のファイルが並ぶ、通常の週替わりレッスン形式の
コースにしか自動対応していない。体験授業やScratch wikiのように「#N」形式ではない
特殊な構成のコースを追加する場合は、スクリプト内の該当セクション
(`taikenLessons` 相当・`scratchWikiLessons`)のようにファイル名と表示順を手で対応付ける
配列をスクリプトに追記する。
